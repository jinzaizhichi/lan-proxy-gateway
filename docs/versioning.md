# 版本规划

## 当前策略

当前公开版本号保持不变，继续沿用现有版本线。

也就是说:

- 不重排已经发布的 `v1.x` / `v2.x`
- 不改历史 tag 指向
- 重点补齐历史 release 的标题、说明和资产说明

这样不会影响:

- 已安装用户的升级路径
- GitHub 上已经公开的版本链接
- `gateway update` 现有版本判断

## 历史版本梳理

定位: 初始公开版。  
重点: 建立局域网透明代理网关主链路，补齐设备接入文档、安装脚本和自动发布流程。

重点: 修复安装和 release 升级链路。

### v1.2.0

定位: Chains 链式代理首版。  
重点: 引入住宅代理链路，服务 AI 场景。

重点: 加入多种代理模式切换能力。

重点: 引入全局扩展脚本能力，把“高级自定义”从模板硬编码里拆出来。

### v2.1.0

重点: 让局域网透明代理的开关更直接。

### v2.2.0

定位: 统一控制台与配置架构。  
重点: TUI 控制台、策略组切换、统一配置分区、release 资产和版本说明体系。

### v2.2.1

定位: 控制台体验修复。  
重点: 修正 TUI 卡顿、状态与配置摘要展示、`console` 重进入口，以及纯命令 / TUI 双模式衔接。

## 之后的使用原则

### Patch

`vX.Y.Z`

适用:

- bugfix
- 文档修正
- 输出文案修正
- release 说明补档

### Minor

`vX.Y.0`

适用:

- 向后兼容的新功能
- 新的命令入口或控制台能力
- 规则系统增强
- chains / skill / 局域网共享增强

### Major

`vX.0.0`

只在下面场景使用:

- 配置结构发生破坏性变化
- 旧命令行为不再兼容
- 运行模型出现重大切换

## 历史 release 的处理原则

为了避免影响已经安装旧版本的用户，历史版本采用下面的方式治理:

1. 不移动已发布 tag 指向
2. 可以补齐 release 标题、描述和缺失资产
3. 可以基于原始 tag 重新编译并上传缺失二进制

这样既能把历史版本整理清楚，也不会破坏现有升级路径。
